# Local Control For ASS-ADE Evolution

This is the local operating contract for rebuild outputs, split evolutions, and
merge siblings. GitHub auto-evolution can keep running; local work should stay
measurable and easy to prune.

## Control Root

Use one dedicated control root:

```text
C:\!aaaa-nexus\!ass-ade-control\
```

The control root is generated by:

```powershell
python C:\!aaaa-nexus\ass-ade-github-latest\scripts\ass_ade_local_control.py init
```

Directory roles:

| Path | Purpose |
| --- | --- |
| `evo/worktrees/` | Active git worktrees only. |
| `evo/lanes/<lane>/` | Lane notes, branch receipts, and active lane metadata. |
| `evo/merge-siblings/` | Candidate sibling sets being compared for a merge. |
| `evo/retired/` | Closed or abandoned lane receipts before archive/delete. |
| `outputs/experiments/` | Short-lived rebuild outputs. |
| `outputs/candidates/` | Outputs attached to a merge candidate. |
| `outputs/releases/` | Pinned release outputs. |
| `outputs/stress/` | Feature-gain and stress-test reports. |
| `archive/` | Frozen outputs that are no longer active. |
| `scratch/` | Disposable local experiments. |
| `inventory/` | Generated local sibling inventories. |

## Names

Use names that encode lineage instead of mood.

| Thing | Pattern | Example |
| --- | --- | --- |
| Branch | `evo/<lane>/<YYYYMMDD>-<slug>` | `evo/memory/20260420-durable-store` |
| Worktree | `wt_<YYYYMMDD>_<lane>_<slug>_<shortsha>` | `wt_20260420_memory_durable-store_6499a168` |
| Output | `rb_<YYYYMMDDTHHMMSSZ>__<lane>__<shortsha>__<slug>` | `rb_20260420T061500Z__memory__6499a168__durable-store` |
| Merge branch | `merge/<YYYYMMDD>/<version>-rc<N>` | `merge/20260420/v0.3.1-rc1` |

Avoid new folders named only `merged`, `unified`, `rebuilt-test`, `latest`, or
`final`. Those are allowed as archived historical names, not new work names.

## Output Metadata

Every rebuild output must contain:

```text
ASS_ADE_OUTPUT.json
MANIFEST.json
CERTIFICATE.json
REBUILD_REPORT.md
```

Stamp an output:

```powershell
python scripts\ass_ade_local_control.py stamp-output C:\!aaaa-nexus\!ass-ade-control\outputs\experiments\<output> `
  --source C:\!aaaa-nexus\ass-ade-github-latest `
  --class experiment `
  --lane quality `
  --branch evo/quality/20260420-example `
  --slug example
```

Validate it:

```powershell
python scripts\ass_ade_local_control.py validate-output C:\path\to\output
```

The schema lives at `schemas/ass_ade_output.schema.json`.

## Split Evolution

1. Start from a clean clone or worktree at GitHub `main`.
2. Create one lane branch per idea.
3. Put each worktree under `evo/worktrees/`.
4. Put each rebuild output under `outputs/experiments/` until it passes gates.
5. Stamp every output with `ASS_ADE_OUTPUT.json`.
6. Promote only passing outputs to `outputs/candidates/`.
7. Create a merge sibling set in `evo/merge-siblings/<date>-<version>-rc<N>/`.
8. Run feature-gain stress comparison before merging.

## Merge Sibling Rule

A merge sibling is not ready unless it has:

- a stamped output
- passing local tests
- a clean source ref
- synced version surfaces
- updated capability registry when claiming a feature gain
- stress-gain report with no severe regression

Compare baseline and candidate:

```powershell
python scripts\ass_ade_local_control.py stress-gain C:\!aaaa-nexus\ass-ade-github-latest C:\path\to\candidate --collect-pytest
```

Store reports in:

```text
C:\!aaaa-nexus\!ass-ade-control\outputs\stress\
```

## Inventory

Generate a local inventory after every active work session:

```powershell
python scripts\ass_ade_local_control.py inventory
```

This writes:

```text
C:\!aaaa-nexus\!ass-ade-control\inventory\latest.json
```

Use the inventory to decide what to prune, archive, or promote. Do not delete a
dirty git checkout or an unstamped candidate output.

## JSON Lifecycle

The local control system has four JSON jobs:

| Job | Command | Output |
| --- | --- | --- |
| Generate inventory | `python scripts\ass_ade_local_control.py inventory` | `inventory/latest.json` |
| Snapshot features | `python scripts\ass_ade_local_control.py snapshot <repo>` | `outputs/stress/*.json` |
| Ingest ledgers | `python scripts\ass_ade_local_control.py ingest-json` | `CONTROL_INDEX.json` |
| Regenerate docs | `python scripts\ass_ade_local_control.py docs-generate` | `docs/capability-matrix.md`, `docs/local-control-status.md`, `docs/mermaid-diagrams.md`, `docs/diagrams/*.mmd` |

The normal one-command refresh is:

```powershell
python scripts\ass_ade_local_control.py refresh --collect-pytest --docs
```

That command updates:

- the latest inventory
- the baseline feature snapshot
- capability registry validation
- the control index
- generated documentation
- generated Mermaid diagrams

## Diff Checking

Compare two local ledgers:

```powershell
python scripts\ass_ade_local_control.py diff-json `
  C:\!aaaa-nexus\!ass-ade-control\inventory\local-inventory-old.json `
  C:\!aaaa-nexus\!ass-ade-control\inventory\latest.json `
  --out C:\!aaaa-nexus\!ass-ade-control\logs\inventory-diff.json
```

The diff command understands inventory files and feature snapshots. Unknown JSON
schemas still get a generic structural diff.

## Registry Validation

Validate feature claims before a merge:

```powershell
python scripts\ass_ade_local_control.py validate-registry
```

The validator checks duplicate IDs, valid statuses, required fields, and whether
local evidence paths exist. Feature growth should be recorded in
`capabilities/registry.json`; generated docs should follow from that file.

## Mermaid Diagrams

Generated diagrams are written by `docs-generate` and `refresh --docs`:

```text
docs/mermaid-diagrams.md
docs/diagrams/json-lifecycle.mmd
docs/diagrams/local-control.mmd
docs/diagrams/capability-status.mmd
```

The diagrams are generated from the same JSON ledgers as the status docs, so
they should not be edited by hand.
